# TextSkill

Стоп-лист AI-штампов на русском, чек-лист для аудита готового текста и восемь скиллов Claude Code, которые пишут и проверяют тексты по этим правилам. Ещё один скилл собирает личный голос автора, чтобы тексты звучали не усреднённо, а как пишешь именно ты.

В корне лежат `writing-rules.md` (ядро универсальных правил) и `audit.md` (чек-лист проверки). Скиллы из `.claude/skills/` подключаются по запросу вроде «напиши пост» или «оцени текст»: каждый читает ядро и добавляет свои правила формата. Скажешь «напиши пост» — включится `write-post` с лимитами Telegram, структурой поста и стоп-листом кликбейта. Для аудита того же поста `write-post` подключится снова, чтобы ловить не только общие паттерны, но и нарушения формата.

## Структура

```
writing-rules.md                # ядро универсальных правил
audit.md                        # методика аудита: шкала и формат (паттерны берёт из ядра)
style-profile.local.md          # личный голос автора — локальный, не в гите (опционально)
.claude/skills/
├── write-post/SKILL.md         # пост для Telegram
├── write-article/SKILL.md      # статья / лонгрид
├── write-readme/SKILL.md       # README.md для GitHub
├── write-docs/SKILL.md         # техническая документация
├── audit-text/SKILL.md         # оценка человечности (детектор AI)
├── audit-docs/SKILL.md         # редакторский аудит документации, 1–10
├── rewrite-text/SKILL.md       # переписать, убрав AI-штампы
└── style-profile/SKILL.md      # собрать профиль авторского голоса
```

## Установка

Вариант 1 — клонировать репо и работать прямо в нём:

```bash
git clone git@github.com:GlebSmolyakov/TextSkill.git
cd TextSkill
claude
```

Вариант 2 — скопировать ядро и скиллы в свой проект:

```bash
cp -r TextSkill/{writing-rules.md,audit.md,.claude/skills} /путь/к/твоему/проекту/
```

Скиллы подключаются автоматически, как только Claude увидит папку `.claude/skills/` в текущем проекте.

## Скиллы

В write-post помимо ядра — лимиты Telegram, шесть типов поста со своей структурой каждый, таблица форматирования и маркировка рекламы по ФЗ-38. Отдельный стоп-лист — кликбейт-приёмы типа «секрет, который изменит...» с пояснением, почему не работает.

У write-article структура похожая, но для длинных текстов: правила открытия, тезис, согласованность уровня глубины через весь текст; для туториалов добавлен отдельный блок про работу с кодом и принцип «статья ≠ книга».

У write-readme правила разнесены по типу проекта — утилита, библиотека, фреймворк или портфолио-кейс, у каждого свой акцент. Внутри также шпаргалка по GitHub-flavored markdown с колаутами `[!NOTE]`, `[!WARNING]` и стоп-лист бэдж-супа, маркетинговых подвалов и скриншотов команд вместо codeblock'ов.

write-docs — для технической документации (справочная статья, описание метода или события API, страница параметра, обзорная страница). У жанра другой регистр, чем у живого текста, поэтому скилл открывается блоком override: в доках нейтральный тон вместо позиции, обязательный вывод вместо открытого финала, не начинать с союзов, числа до десяти словами. Продуктовую специфику (термины, разметку, имена файлов меню) скилл не держит — она остаётся в локальной редполитике репозитория.

audit-text устроен как методика проверки на AI: пошаговый проход с разделением паттернов на критичные/средние/мелкие, шкала 1–10 с маркерами. audit-docs — отдельная методика для документации: 8 шагов сверху вниз с упором на фактическую точность, та же шкала 1–10, но цель не «звучит ли как AI», а редакторское качество. В rewrite-text — пять слоёв обработки (лексика → синтаксис → ритм → структура → позиция) с параметром глубины; для доков он подтягивает write-docs и уважает его override. На выходе — одна версия, без «или можно ещё так».

style-profile стоит особняком: он не пишет и не проверяет, а собирает голос автора. Задаёт несколько вопросов, разбирает твои образцы и пишет `style-profile.local.md` в корень. Дальше write-post, write-article, write-readme и rewrite-text читают этот файл и пишут в твоём голосе, а не в усреднённом. Порядок слоёв: ядро снимает машинность → скилл задаёт формат → профиль задаёт голос; при конфликте по существу выигрывает ядро (профиль не может «разрешить штампы»). Файл локальный, в гит не идёт (`*.local.md` в `.gitignore`), у каждого автора свой. На документацию профиль не распространяется — там нейтральный регистр.

## Использование

В Claude Code открой проект и напиши что нужно, а скилл подхватится по описанию:

- «напиши пост про провальную рекламную кампанию» → активируется `write-post`, задаст пару уточняющих вопросов (канал, тон, тип поста), выдаст пост по правилам Telegram-формата.
- «опиши метод crm.item.add» → `write-docs` пишет справочную страницу в нейтральном доковом регистре (override ядра внутри).
- «проверь этот текст» + текст → `audit-text` прогоняет по чек-листу, возвращает балл 1–10 и список паттернов по тяжести.
- «проверь эту документацию» + текст → `audit-docs` идёт по 8 шагам с упором на факты, возвращает балл 1–10 и findings.
- «перепиши, убери AI-паттерны» + текст → `rewrite-text` спрашивает глубину переработки и идёт по пяти слоям.
- «настрой мой стиль» / «пиши как я» → `style-profile` задаёт несколько вопросов, разбирает образцы и собирает `style-profile.local.md`. После этого скиллы живого текста пишут в твоём голосе.

> [!NOTE]
> Для поста audit-text и rewrite-text дополнительно подключают write-post, для статьи — write-article, для документации — write-docs. Аудит учитывает не только запреты ядра, но и правила формата. У документации регистр инвертирован: audit-docs не штрафует за нейтральный тон и отсутствие позиции — для доков это норма.

## Если ты не в Claude Code

Промты для обычного чата.

**Пост для Telegram**
```
Используй правила из writing-rules.md как базу.
Дополнительно — .claude/skills/write-post/SKILL.md.

Тема: [тема]
Канал: [о чём канал, кто читает]
Тон: [разговорный / деловой / личный]
Тип поста: [личный / новостной / экспертный / рекламный / опрос / анонс]
```

**Статья**
```
Используй правила из writing-rules.md как базу.
Дополнительно — .claude/skills/write-article/SKILL.md.

Тема: [тема]
Тезис: [главная мысль]
Аудитория: [кто читает, что уже знает]
Объём: [примерно слов]
Тип: [эссе / аналитика / туториал / обзор / технические «кишочки» / DIY]
```

**Документация**
```
Используй writing-rules.md как базу.
Дополнительно — .claude/skills/write-docs/SKILL.md (сначала примени блок override регистра).
Если есть локальная редполитика репозитория с продуктовыми терминами — примени поверх.

Тип страницы: [справочная статья / описание метода или события / страница параметра / обзорная]
Тема: [что описываем]
```

**Переписать**
```
Используй writing-rules.md.
Если текст — пост, добавь .claude/skills/write-post/SKILL.md.
Если статья — .claude/skills/write-article/SKILL.md.
Если документация — .claude/skills/write-docs/SKILL.md (его override отменяет правила ядра про позицию, финал, союзы, числа).

Перепиши: убери AI-паттерны, смысл и структуру сохрани.
Глубина: [лёгкая / средняя / глубокая]

Текст:
---
[текст]
---
```

**Аудит живого текста (детектор AI)**
```
Используй паттерны из writing-rules.md как чек-лист.
Дополнительно — методика и шкала из audit.md.

Тип текста: [пост / статья / письмо / лендинг]

Текст:
---
[текст]
---
```

**Аудит документации**
```
Используй writing-rules.md и .claude/skills/write-docs/SKILL.md как правила.
Методику и шкалу 1–10 бери из .claude/skills/audit-docs/SKILL.md.

Тип страницы: [справочная статья / описание метода / страница параметра / обзорная]

Текст:
---
[текст]
---
```
