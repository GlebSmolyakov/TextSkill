Прочитай `writing-rules.md` из корня проекта — база универсальных правил. Но документация — другой жанр, чем живой авторский текст: голос нейтральный и объясняющий, а не личный с позицией. Поэтому сначала примени блок override ниже, он отменяет часть правил ядра. Всё остальное из ядра (борьба с канцеляритом, водой, рекламщиной, номинализацией, «главное вперёд», ритм, типографика) действует как есть.

---

## Override ядра — в документации наоборот

Эти правила ядра в документации **не применяются** — у жанра другой регистр:

| Ядро (живой текст) | Документация |
|---|---|
| Займи позицию, пиши от «я» | Нейтральный безличный тон. Не давать оценок сложности, удобства, полезности. Не писать за читателя, что ему удобно |
| Не заканчивай подводкой, оставь открытым | Если формат предполагает завершённый текст — заключение обязательно. Текст не обрывается внезапно: в конце вывод, который следует из содержания |
| Начинай предложения с «И», «Но», «А» | Не начинать с союзов без смысловой необходимости. «Потому что», «а также» в начале — частый признак неудачной связки с прошлым предложением |
| Числа всегда цифрами | До десяти включительно — словами, после десяти — цифрами. В противопоставлении и длинных перечнях — цифрами. Не наращивать окончания: «третий», не «3-й» |
| Разговорные частицы, смена регистра | Ровный нейтральный стиль без скатывания в разговорный или избыточно научный |

Доковые конвенции, которых в ядре нет:
- Всегда `е`, не `ё`.
- «вы» — только со строчной буквы.
- Полные формы вместо сокращений: «и так далее» (не «и т.д.»), «и другое» (не «и др.»), «то есть» (не «т.е.»). Для мегабайт — «Мб».
- Путь пользователя — только курсивом, шаги через разделитель `>`, без дублирующего жирного/цвета.
- Жирный — только лид в начале абзаца, не произвольный акцент внутри строки.
- Не ставить двоеточие перед скриншотом.

## Что считать хорошей страницей

Текст отвечает на два вопроса: **что можно сделать после прочтения и зачем**. Структура читается по диагонали — основное понятно при беглом просмотре, по заголовкам прослеживается логика.

- Заголовок содержит главную мысль страницы, по возможности с глаголом. Без вопросительных и восклицательных знаков.
- Подзаголовок выражает мысль своего раздела и согласован с заголовком.
- Заголовки и подзаголовки — обычная русская норма прописных: лишние слова с заглавной не поднимать (не Title Case).
- Длину не гнать ради объёма. Тема объяснена без потери смысла — лишний абзац не добавлять. Остаются вопросы «что это», «зачем», «что дальше» — расширить.

## Введение и заключение

- Введение — без воды, рекламных конструкций и абстракций. Риторический вопрос во введении переписать в повествование.
- Для обзорных/индексных страниц работает вводный блок из двух абзацев: первый — что это за раздел или объект, второй — как он используется на практике.
- Если тему можно приземлить коротким реалистичным примером — дать его сразу после определения.
- Если в разделе две модели, два режима или два уровня настройки — развести их уже во введении или в отдельном сценарном блоке.
- Заключение обязательно там, где предполагается завершённый связный текст. Вывод следует из содержания, а не лепится для галочки.

## Абзацы и предложения

- Абзац — одна основная мысль, начинается с главного («главное вперёд»).
- Длинные абзацы (8–9 строк и больше) — делить. Чередовать длину абзацев.
- Одно предложение — одна мысль. Предпочитать простые предложения сложным.
- Предложение около трёх строк и больше — проверить, не вместило ли две-три мысли; при необходимости разбить. Если трудно прочитать вслух на одном вдохе — сократить.
- Подлежащее + сказуемое + остальное. Избегать инверсии, которая уводит главное в конец.
- Сложноподчинённое допустимо, пока прозрачно. Накопилось несколько придаточных — упростить или разбить.
- Канцелярит — убирать: страдательный залог → активный, цепочки существительных и отглагольные → глаголы, неопределённо-личное → понятное действующее лицо.

## Списки

- Список — для шагов, вариантов или чтобы разгрузить перегруженное перечисление из сплошного текста.
- Пункты согласованы между собой и с вводной фразой: одна часть речи, ответ на один вопрос.
- Нумерованный — для последовательных действий по порядку; маркированный — для вариантов, где не обязательно всё.
- Неполная вводная фраза с двоеточием → пункты со строчной. Пункты-полные предложения и любой нумерованный список → с заглавной.
- Общее начало всех пунктов выносить во вводную фразу. Три и более шага подряд с одинаковым зачином («Если нужно») — разбавить синонимом («Чтобы»).
- Один уровень вложенности, не больше.

## Ссылки

- Текст ссылки сам объясняет, куда ведёт. Никаких «нажмите сюда», «смотрите тут».
- Если ссылка ведёт на статью, сценарий или туториал и без этого фраза звучит обрывочно — назвать тип прямо.
- В блоке с несколькими сценариями оформлять ссылки симметрично.

## Термины и повторы

- Термины — последовательно, без смешения названий одного объекта. Повтор термина лучше неочевидного синонима: точность важнее разнообразия.
- Повторы глаголов чаще лишние — заменять точным синонимом или перестраивать фразу.
- Если рядом повторяются служебные «нужно», «использовать», «используется» — перестроить.
- Не использовать публицистические слова, нехарактерные для нейтральной речи («злоумышленник», «хакер») — описывать само действие, ошибку или ограничение системы.

## Структура справочника по API (общая)

Применимо к API-документации в целом; продуктовые термины и разметку держи в локальном файле репозитория, не здесь.

- **Индексная (обзорная) страница** отвечает на четыре вопроса: что это за раздел, как начать работу, с какими объектами связан, какие методы есть.
- В таблице методов писать, **что делает** метод: описание начинать сразу с глагола, не со слова «метод». Для событий — формулировка «При …».
- Параметр, впервые появившийся в шаге или абзаце, не должен возникать без короткого пояснения: что означает, где взять значение, по возможности пример.
- Для обзорных страниц параметр объяснять в формате: что это → где используется → как получить.
- Если нужен идентификатор — называть конкретный метод, которым его получают, а не отправлять «в соседний раздел».
- Warning-плашку — только для реального ограничения или поведения, которое читатель поймёт неправильно; не как способ выделить важную мысль.
- Не дублировать одну мысль в нескольких блоках. Общие технические детали уровня «list использует start для постраничной навигации» не выносить как особенность раздела.

## Параметры

Если из запроса не очевидно — спроси:
- **Тип страницы** (справочная статья / описание метода или события / страница параметра / обзорная (индексная) страница / гайд).
- **Продукт и его конвенции** — есть ли локальная редполитика репозитория с продуктовыми терминами и разметкой (их применяй поверх этого скилла).

## Фактическая точность

- Не приписывать методу поведение, которого у него нет. Метод чтения только возвращает сохранённое — так и формулировать, без додуманной логики преобразования.
- Если реализация подтверждает частный случай — не расширять до общего правила без подтверждения в коде или документации.
- Любое искажение факта — критичная ошибка, не стилистическая.

## Перед выводом

Прочитай страницу «вслух»: ловятся канцелярит, спотыкающийся ритм, предложение с двумя мыслями. После — одна-две строки: тип страницы и какие продуктовые конвенции репозитория надо было учесть (если применял локальный файл).
