## Три фазы — строго по порядку

Не переходи к следующей фазе, пока не завершена предыдущая. Не пиши текст до конца фазы 2.

**Фаза 1. Голос.** Если в корне есть `style-profile.local.md` — прочитай его. README не личный текст, но авторский регистр задаёт тон описаний и примеров.

**Фаза 2. Фактура.** Задай эти вопросы. Без конкретных ответов открытие выйдет обобщённым, а инструкции — неточными:
- Что конкретно лежит в репо? Файлы, команды, структура.
- Точная команда установки (копипастом)?
- Минимальный пример использования с реальными значениями, не `foo/bar`?
- Открытие: твоя первая строка, конкретная ситуация — или писать самому?

**Фаза 3. Генерация.** Только после получения ответов — писать по правилам формата ниже.

## Параметры

Если из запроса не очевидно — спроси:
- **Что за проект** (CLI / библиотека / приложение / фреймворк / репо с правилами или статьями / документация / портфолио или pet-project для найма)
- **Аудитория** (внутренняя команда / внешние разработчики / новички / опытные / HR и тимлиды, если это портфолио)
- **Размер проекта** (однофайловая утилита → короткий README; средний → стандартный; большой фреймворк → полный + ссылки на доп. файлы)
- **Структура файлов** (есть ли скелет, на который надо сослаться)
- **Открытие** — твоя первая строка, конкретная ситуация или «напиши сам». Без якоря открытие выйдет обобщённым. Если не дал — спроси.

## На какие вопросы отвечает README

Базовый чек-лист — на какие вопросы должен ответить README. Не все обязательны, но пройди по списку и осознанно реши, что нужно для конкретного проекта:

1. **Что это** — одной-двумя строками, без «X — это инструмент для Y».
2. **Какие проблемы решает / для чего создано** — конкретные кейсы, не абстрактные «повышает эффективность».
3. **Чем отличается от похожих решений** — если в нише есть аналоги. Только конкретные факт-чекаемые отличия, не «самый быстрый» без бенчмарка.
4. **Как установить / подключить** — команды копипастом, без «следуйте этим шагам».
5. **Как пользоваться** — минимальный рабочий пример. Реальный, не `foo` / `bar` / `baz`.
6. **Как принять участие в развитии** — если открытый проект. Можно явно «contributions не принимаются», если автор не хочет помощи.
7. **Кто разрабатывает** — имена / организация / ссылка на команду. Для open source важно — повышает доверие к проекту.

Минимум для большинства проектов — пункты 1, 4, 5. Остальные — по делу.

Пустые секции хуже отсутствующих. Не копируй стандартный шаблон, если разделы пустые.

## Структура заголовков на GitHub

- `# H1` — только название проекта, один раз
- `## H2` — основные разделы (Установка, Использование, Разработка, Участники)
- `### H3` — подразделы внутри H2

H4+ на GitHub визуально почти не отличаются от обычного текста, поэтому держи глубину до H3.

## Принципы

### Открытие — без «это проект для»
Самая частая ошибка — «MyTool — это инструмент для X, который помогает Y». Это самоопределение через пользу (запрещено в writing-rules.md). Альтернативы:
- Сцена: «Открываю чужой пост и вижу «безусловно, данный»...» — показ через ситуацию использования.
- Прямая констатация: «Стоп-лист AI-штампов на русском и человечные альтернативы.» — что лежит, без объяснения зачем.
- Минимум: одна короткая строка о содержании, без «помогает» и «позволяет».

Но и «чистая» альтернатива тоньше скатывается в тот же питч. Что отдельно ловить в открытии README:
- **Балансный контраст** вроде «руками долго, а через ИИ шаблонно» — симметричная пара «X плохо так, Y плохо эдак» читается как слоган, а не как живая жалоба.
- **Проблема словами самого продукта** — если боль описана теми же терминами, которыми ты хвалишь решение («шаблонно», «вода», «штампы»), выходит реклама самого себя.
- **Ровная пара «что это + проблема»** — два гладких предложения, выстроенные как слайд питч-дека. Живое открытие кривее: начни с конкретного случая или своими словами, как рассказал бы коллеге (см. «Пиши его словами» в ядре).

### Размер — по делу
- Микро-утилита (одна функция) — 10–30 строк
- Средний проект — 100–300 строк
- Фреймворк — больше, но с разбивкой по дополнительным `.md` файлам (`docs/`, `CONTRIBUTING.md`, `ARCHITECTURE.md`)

Не растягивай README в маркетинговый лендинг.

### Один scope
README отвечает на «что» и «как». Не на «почему именно так» (это в `ARCHITECTURE.md` или в блоге) и не на «как это работает изнутри» (это в комментариях кода).

### Скан, а не чтение
README сканируют, а не читают подряд:
- Шапка → «о чём это»
- Codeblock с командой установки
- Codeblock с примером использования
- Дальше — если читателю интересно

Размести главное так, чтобы при беглом скролле картина уже сложилась.

### Команды — точные
Команды копипастятся. Если поменялся флаг — поменяй в README в том же коммите, что меняешь сам флаг. Устаревшая команда хуже её отсутствия.

### Очевидное для автора — не очевидно для читателя
Ты живёшь с проектом неделями. Каждый шаг внутри головы, каждая зависимость подразумевается. Читатель видит впервые.

Стэнфордский эксперимент: участники настукивали пальцами по столу мелодию популярной песни. По их оценке, угадать должны были 50% слушателей. Реально угадывали 2,5%. Авторы переоценивают, насколько хорошо доносят то, что внутри их головы.

Применительно к README: если в инструкции по установке не сказано «нужен Node.js 18+» — для тебя самоочевидно, а у читателя на этом шаге всё ломается. Если в примере использования есть переменная окружения, которая «понятно откуда» — её надо явно объяснить.

Проверь: дай README прочитать тому, кто не работал над проектом. Каждое «а откуда взять X?», «а как это запустить?» — пропущенный шаг.

## Если это портфолио / pet-project для найма

Отдельный жанр со своими акцентами. Аудитория — HR и тимлид, которые часто не разработчики. Цель — показать, как мыслишь и оформляешь, а не «как использовать в проде».

### Акценты, которые добавляются к базовому чек-листу

- **Value statement в первой строке.** Что проект делает для пользователя одной фразой. Не «test project», не «my app», не «пет-проект на React» — это сразу минус балл. «HabitTrack — трекер привычек с аналитикой и напоминаниями» лучше, чем «HabitTrack» в одиночку.
- **Стек с пояснением «где и зачем».** Не «React, Node, MongoDB», а «React + Redux — фронтенд с управлением состоянием; Node.js + Express — API; MongoDB — хранение пользовательских привычек».
- **Раздел «Чему научился / выводы».** Что было сложным, что переделывал, какие компромиссы выбирал. Показывает рефлексию — для HR это редкий сигнал зрелости.
- **Скриншот UI или GIF-демо.** В отличие от скриншота консольной команды — это нормально и полезно (см. ниже). Для веб-приложения — главный экран + скрин ключевого сценария.
- **Демо-ссылка** — если есть рабочий деплой, ставь её сразу в шапку. Hosted demo > screenshot > описание.
- **Расшифровка специфического жаргона.** HR может не знать, что такое SSR, GraphQL, Redux-Saga. Либо избегай терминов, либо в одну строку поясни.

### Акценты по роли

- **Frontend** — UX и визуал на первом плане: скриншоты, дизайн-система, Lighthouse-оценка, если есть.
- **Backend** — архитектура и данные: схема БД (PlantUML, dbdiagram.io), OpenAPI/Swagger, описание безопасности (auth, CORS, rate limits).
- **Fullstack** — интеграция и пайплайн: CI/CD (GitHub Actions, Jenkins), деплой (Docker, AWS, Render), автоматические тесты.
- **Дизайн / frontend с дизайн-уклоном** — Design-to-Code: ссылки на макеты Figma, описание принципов вёрстки, дизайн-токены.

### Чего в портфолио-README быть не должно (даже несмотря на советы из интернета)

- Эмодзи в заголовках («🚀 Features», «🎉 Quick Start»). В интернете советуют — устаревает быстрее остального и раздражает технического читателя.
- Десяток бэджей в шапке. Build status, coverage, license — нормально (2–3). 15 значков подряд — нет.
- Лирические отступления «Да, я люблю темную тему!» как способ «показать личность». Личность видна по делу — по выбору проекта, по тому, как написан раздел «чему научился». Не нужно её декларировать.

## GitHub-flavored Markdown — что использовать

(Семантика заголовков — выше, в разделе «Структура заголовков на GitHub».)

**Ссылки.** `[текст](url)` или `<url>` для автоматических. Относительные ссылки между файлами репо: `[архитектура](docs/architecture.md)`. Якоря к заголовкам в том же файле: `[#installation](#installation)`.

**Изображения.** `![alt](url)`. Для контроля размера — HTML: `<img src="logo.png" width="200">`.

**Код.**
- Инлайн: \`flag\`, \`functionName()\`
- Блок с подсветкой синтаксиса:

````
```bash
npm install textskill
```

```python
from textskill import audit
audit(text)
```
````

**Таблицы.**

```
| Опция   | По умолчанию | Описание         |
|---------|--------------|------------------|
| --port  | 8080         | Порт сервера     |
| --debug | false        | Включить отладку |
```

**Цитаты и колауты (GitHub-специфичное).**

```
> Обычная цитата.

> [!NOTE]
> Дополнительная информация.

> [!TIP]
> Полезная подсказка.

> [!IMPORTANT]
> Критично для понимания.

> [!WARNING]
> Предупреждение об опасном поведении.

> [!CAUTION]
> Что-то реально опасное.
```

Эти колауты рендерятся на GitHub в цветные плашки. Не злоупотребляй — один-два на README достаточно.

**Списки.**
- Маркированный: `-` (предпочтительно) или `*`
- Нумерованный: `1.` `2.` `3.`

Не делай вложенные списки глубже двух уровней — теряется читаемость.

**Горизонтальная линия.** `---` — между крупными секциями. Не злоупотребляй.

**Жирный / курсив.** `**жирный**`, `*курсив*`. Не выделяй жирным целые предложения.

## Чего в README никогда нет

**Маркетинговый пафос:**
- «Революционный», «единственный в своём роде», «лучший выбор»
- «Просто и легко!» без демонстрации — покажи код за две строки, поверят
- «Молниеносно быстрый», «production-ready» (если это не подтверждено бенчмарками)

**Шум в шапке:**
- Бэдж-суп: 15 значков в ряд (build, coverage, npm, license, downloads, stars, twitter, discord, sponsors...). Оставь 2–3 действительно важных.
- Эмодзи в заголовках: «🚀 Features», «🎉 Quick Start», «⭐ Why us». Раздражает и устаревает быстрее остального.

**Маркетинговые подвалы:**
- «Made with ❤️ by ...»
- «Star us on GitHub!»
- «Don't forget to ⭐ this repo if it helped!»
- «Надеюсь, было полезно. Звёздочка приветствуется!»

**Технический шум:**
- **Скриншот командной строки** вместо вывода в codeblock — команду нельзя скопировать, нельзя поискать в Ctrl+F, не работает screen reader. Для CLI всегда показывай вывод текстом в блоке.
- (Скриншот UI приложения — наоборот, нужен. Это другой случай.)
- Огромное оглавление в README на 200 строк
- «Coming soon» на половину пунктов — если не готово, не пиши
- Описание установки в шести вариантах (npm / yarn / pnpm / bun / brew / docker) для крошечной утилиты — один путь главный, остальные одной строкой

**Содержательный шум:**
- Длинное «Why this exists» во вступлении (если только это не reflexion-доку, а тут он не место)
- Раздел Features на 15 пунктов, где половина повторяет половину другими словами
- Подробный API-референс в README — вынеси в отдельный файл `docs/api.md`
- Шаблонные «Contributing» и «License» для личного одноразового скрипта

## Частые ошибки

Помимо «чего в README никогда нет» выше:

- **Одушевил репо** — «знает», «поддерживает», «понимает», «умеет», «помогает», «обеспечивает». Используй «есть», «лежит», «работает с» или перечисление через тире.
- **Безымянная ссылка** — текст ссылки должен сам объяснять, куда ведёт. «Click here», «здесь», «тут», «см. документацию» не ищутся по Ctrl+F и не работают со screen reader. Пиши `[гайд по установке](...)`, не `установка [здесь](...)`.
- **Оставил плейсхолдеры из шаблона** — «TODO: описать», «Lorem ipsum», «replace this section».
- **Неправильная подсветка кода** — «```scss» для CSS-блока. Используй точную (`css`, `python`, `bash`) или никакую.
