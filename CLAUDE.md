# CLAUDE.md

TextSkill — набор правил и скиллов, которые заставляют тексты на русском звучать по-человечески, а не как генерация AI. Это не код: ядро правил + автозапускаемые скиллы Claude Code.

## Единый источник правды

- **`writing-rules.md`** — единственный источник паттернов: запрещённая лексика, синтаксис, структура, типографика, ритм, позитивные правила. Любое новое правило про живость текста идёт сюда.
- **`audit.md`** — только методика аудита (карта категорий + шкала + формат отчёта). **Не дублируй сюда стоп-листы.** Паттерны он берёт из ядра. Если переписываешь промт для обычного чата — ссылайся на ядро, а не копируй его.
- Форматные правила (структура поста, статьи, README) живут в соответствующем `.claude/skills/*/SKILL.md`, не в ядре.

## Маршрутизация скиллов

Скиллы срабатывают автоматически по описанию во фронтматтере. Логика выбора:

- «напиши пост / для тг» → **write-post** (короткий текст, до ~5 абзацев).
- «напиши статью / лонгрид / эссе» → **write-article** (от нескольких сотен слов).
- «напиши / обнови readme» → **write-readme**.
- «напиши документацию / опиши метод / API / справочную страницу» → **write-docs**.
- «проверь / оцени текст, насколько живой» → **audit-text** (детектор AI).
- «проверь / оцени документацию» → **audit-docs** (редакторское качество + факты, шкала 1–10).
- «перепиши / оживи / убери AI» → **rewrite-text**.
- «настрой / определи мой стиль», «пиши как я» → **style-profile** (интервью + разбор образцов → локальный `style-profile.local.md`).

Граница audit ↔ rewrite: «оцени, что не так» — это audit (диагноз, балл). «Сделай лучше» — это rewrite (новый текст). Если из запроса неясно — спроси, что нужно: разбор или переписать.

Граница audit-text ↔ audit-docs: разная цель. **audit-text** ловит признаки AI (предсказуемость, ритм) в живом тексте. **audit-docs** оценивает редакторское качество и фактическую точность документации — и не штрафует за нейтральный безличный тон, который для доков норма.

Каждый скилл **сначала читает `writing-rules.md`**, потом применяет свои форматные правила. `audit-text`, `audit-docs` и `rewrite-text` дополнительно подтягивают форматный скилл под тип текста: пост → write-post, статья → write-article, README → write-readme, документация → write-docs.

**Локальный авторский голос.** Поверх ядра и формата работает третий, необязательный слой — `style-profile.local.md` в корне. Если файл есть, скиллы живого текста (write-post, write-article, write-readme, rewrite-text) пишут в голосе автора, описанном там. Хук — в `writing-rules.md` (раздел «Авторский голос»), порядок слоёв: ядро снимает машинность → формат задаёт структуру → профиль задаёт голос; при конфликте по существу выигрывает ядро. Собирает профиль скилл **style-profile**. Файл локальный (`*.local.md` в `.gitignore`), у каждого автора свой, в гит не идёт. На документацию слой не распространяется — там нейтральный регистр.

**Документация — особый регистр.** write-docs содержит блок override: часть правил ядра в доках инвертирована (нейтральный тон вместо позиции, обязательный вывод вместо открытого финала, не начинать с союзов, числа до десяти словами). Любой скилл, работающий с докой, применяет ядро **после** этого override, а не вместо. Продуктовая специфика (термины, разметка, имена файлов меню) в TextSkill не живёт — она остаётся в локальной редполитике конкретного репозитория, write-docs только ссылается на неё.

## Чего пока нет

Генераторов под **письмо, продающий лендинг, описание продукта** нет. `audit`/`rewrite` знают эти типы, но написать с нуля такой текст профильным скиллом нельзя — запрос уйдёт в общий чат. Если нужен такой формат — это новый скилл, а не правка существующего.

Документация уже покрыта: **write-docs** (генерация) и **audit-docs** (редакторский аудит). Но они держат только жанр; продуктовая редполитика конкретного репозитория — отдельный локальный файл, который их дополняет.

## Дисциплина правок

- **Не раздувай списки.** Стоп-листы уже покрывают основные каталоги AI-маркеров (Википедия RU/EN, инфостиль). Ещё одна строка почти не двигает качество, но грузит контекст. Лучше править формулировки в ядре, чем добавлять пункты.
- **Держи примеры краткими.** В скиллах с примерами (write-post, audit-text, rewrite-text) образец якорит планку — но один и компактный.
- **Синхронизация не нужна** — раз паттерны в одном месте, дублей нет. Если заметил копию стоп-листа где-то ещё — это баг, убери.

## Допущение путей

Скиллы читают `writing-rules.md` и `audit.md` из корня проекта. Копируешь скиллы в другой репозиторий — тащи ядро вместе с ними, иначе скиллы сломаются.
